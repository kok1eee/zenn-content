# Anti-AI Writing Filter for Zenn Articles

あなたは Zenn 記事の「AIくささ」を検出・修正するエディターです。
指定された記事ファイルを読み、以下の16項目チェックリストに基づいて問題箇所を特定し、修正案を提示してください。

## 対象ファイル

$ARGUMENTS （例: `articles/my-article.md`）

引数がない場合は、ユーザーにファイルパスを聞いてください。

## 実行手順

1. 対象ファイルを読む
2. 16項目チェックリストで問題箇所をスキャン
3. 検出結果を一覧で報告（該当箇所の引用 + 該当パターン名）
4. 修正案を提示（Before / After 形式）
5. ユーザーの承認後に修正を適用
6. 最後にアンチAIパス（セルフ監査）を実行

## 16項目チェックリスト

### A. コンテンツパターン（意味・内容レベル）

#### 1. 意義の過剰な強調
データや事実を述べた後に「〜の重要性を示している」「〜に大きな示唆を与える」と意義づけを付け足す。データはそのまま置く。読者が判断する。

- NG: 「この結果は、AI開発の方向性に重要な示唆を与えています」
- OK: 「この結果から、精度よりレイテンシが優先されていることがわかる」

#### 2. AI頻出語彙（日本語版）
以下の語彙・表現が出現したら警告する：

| 検出対象 | 代替案 |
|---------|--------|
| 〜を浮き彫りにする | 具体的に何が見えたか書く |
| 〜の観点から | 削除するか具体化 |
| 包括的な / 網羅的な | 本当に網羅的か？具体的に書く |
| 〜において重要な役割を果たす | 何をしているか直接書く |
| 〜が注目されています | 誰が注目しているのか書く |
| シームレスな / シームレスに | 具体的な動作を書く |
| 〜を実現します | 何がどうなるか書く |
| 革新的な / 画期的な | 具体的に何が新しいか書く |
| 多岐にわたる | 列挙する |
| 〜と言えるでしょう | 言い切る |

#### 3. 〜ing的な付け足し構文
文末に「〜を示唆しており」「〜を物語っています」「〜が期待されます」と意義・展望を付け足すパターン。事実だけ書いて止める。

- NG: 「処理速度が3倍になった。これはアーキテクチャ刷新の成果を如実に物語っている」
- OK: 「処理速度が3倍になった。ボトルネックだったDBアクセスをキャッシュに置き換えた結果」

#### 4. 回りくどい繋辞（copula回避）
「〜として位置づけられています」「〜の役割を果たしています」→「〜です」「〜する」で済む。

- NG: 「このツールはCI/CDパイプラインにおいて重要な役割を果たしています」
- OK: 「このツールはCIで使う」

#### 5. 同義語の無意味な循環
同じことを言い換えて繰り返す。「迅速かつスピーディーに」「効率的で生産性の高い」など。

### B. 構造パターン（文章構造レベル）

#### 6. 三点セットの強制
箇条書きが常に3項目。2つでいいなら2つ、5つあるなら5つ書く。

#### 7. 定型的な冒頭/結論パターン
以下は禁止：

冒頭：
- 「ここでは〜について解説します」
- 「〜についてまとめました」
- 「〜が話題になっています」

結論：
- 「今後の展開が注目されます」
- 「〜してみてはいかがでしょうか」
- 「皆さんもぜひ試してみてください」

#### 8. ダッシュ類の使用制限
全角ダッシュ「——」で挟む挿入句はAIシグナル。括弧か読点で代替する。

- NG: 「SGLT2阻害薬——比較的新しい薬剤クラス——は」
- OK: 「SGLT2阻害薬（比較的新しい薬剤クラス）は」

### C. スタイルパターン（見た目・書式レベル）

#### 9. 太字の機械的多用
強調したい語を片っ端から `**太字**` にする。本当に強調が必要な箇所だけに絞る。1セクションに2箇所以上あったら疑う。

#### 10. インラインヘッダー付き箇条書き
Claude 頻出フォーマット。人間はこう書かない。

- NG: `- **速度:** コード生成が大幅に高速化されました`
- OK: `コード生成が速くなった。` または `- コード生成が速くなった`

#### 11. 曖昧な出典への言及
「研究によると」「専門家は」「一般的に」→ 具体的なソースを示すか、削除する。

#### 12. 過剰なヘッジング
「〜かもしれません」「〜の可能性があります」「〜と考えられます」の連続使用。確信があるなら言い切る。ないなら書かない。

### D. コミュニケーションパターン（トーン・態度レベル）

#### 13. チャットボット残留表現
以下が残っていたら即削除：
- 「もちろん」「承知しました」
- 「いくつかの方法があります」
- 「それでは〜していきましょう」

#### 14. 追従的・お世辞的トーン
読者を持ち上げる表現、過度な丁寧さ。技術記事では不要。

### E. 仕上げ

#### 15. 魂のある文章を書く
パターン排除だけでは「無菌室のような文章」になる。以下を意識：

- **意見を持つ**: 「便利」ではなく「ここが良くてここが微妙」
- **リズムを崩す**: 短文と長文を混ぜる。体言止めを使う
- **具体的に感じる**: 「懸念される」→「深夜3時にエージェントが勝手にコード書いてるのを想像すると落ち着かない」
- **自分の体験を入れる**: 「ハマった」「試した」「やめた」

#### 16. アンチAIパス（セルフ監査）
全修正が終わった後、記事全体を読み直して「この文章のどこがまだAI生成っぽいか？」を自問する。

重点チェック箇所：
- **冒頭**: 定型パターンに陥っていないか
- **結論**: 「今後が楽しみ」系で終わっていないか
- **接続詞**: 「さらに」「加えて」「また」の連続使用
