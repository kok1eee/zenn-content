## はじめに

> 業務自動化Pythonエンジニア → Rust でバイブコーディングに目覚めた人。

前回の記事で「RAG パイプラインを捨てて `claude -p` に Grep させたらナレッジ Bot が動いた」と書いた。

https://zenn.dev/tazawa_masayoshi/articles/knowledge-chatbot-agentic-search

あの確信には裏がある。本業で **Slack に書くだけでコードが書かれ、PR が出る**タスク自動化エージェントを Rust で作っていた。毎日 `claude -p` を使い倒す中で「このハーネスなら RAG パイプラインも要らない」と気づいた。ナレッジ Bot は副産物だった。

今回はその本体の話。**自律エージェントを作らなかった理由**と、代わりに選んだ設計思想について書く。

:::message
社内チーム向けに作ったツールのため、コードは非公開。設計思想とアーキテクチャの話が中心です。
:::

## AI エージェントは「自分の生産性を上げるもの」だと思っていませんか

[OpenClaw](https://github.com/openclaw/openclaw) や [gstack](https://github.com/garrytan/gstack) は「自分が Claude に指示して、自分のタスクを進める」ツールだ。主語は自分。

自分が作ったのは違う。**同僚が Slack に書いたら、勝手に動く**。

```
同僚「この店舗のサブカテゴリ追加して」
  ↓
（自分は何もしていない）
  ↓
Bot「完了しました。image_mappings.yaml を更新し、画像を配置しました」
```

本人は別の仕事をしている。会議中かもしれない。寝ているかもしれない。それでも依頼は処理される。だから **ambient**（環境的・常駐的）。

これは自分の生産性を上げるツールではなく、**チームの依頼を勝手にさばくシステム**だ。

## 全体像

```
入口が違うだけで、中の処理は同じ

  Slack メッセージ ─┐
                    ├→ classify → 明確 → そのまま実行 → PR
  Asana タスク    ─┘            → 曖昧 → Slack ラリーで要件確定 → 実行
                                → 詰まった → manual（人間が terminal で対応）
```

3つの入口がある。

| 入口 | 処理 |
|------|------|
| Slack ops（質問・依頼） | 自動回答 or 要件ヒアリング → タスク昇格 |
| Slack ops（定型作業） | スキル実行 → Slack 返信で完了 |
| Asana タスク | 分類 → 実行 → PR → CI |

コーディングだけでなく、PM スキル（朝会ブリーフィング、タイムボクシング、停滞検知、Google Calendar 連携）も備えている。

## なぜ OpenClaw を使わなかったか

最初に検討したのは [OpenClaw](https://github.com/openclaw/openclaw)。マルチチャンネル対応の汎用パーソナルアシスタントとして完成度が高い。でも、チーム業務自動化という用途では合わなかった。

**セキュリティ**: OpenClaw はデフォルトでエージェントがホスト上でフルアクセスで動作する。チーム Slack からの任意メッセージがコード実行に繋がる経路を持つ。自分たちの用途では、入力を Asana タスク（承認済み）と ops 管理者に限定したかった。攻撃面が広すぎる。

**コスト**: OpenClaw は Gateway が常時起動し、セッションが持続する。Heartbeat も30分ごとに LLM を呼ぶ。タスクがあるときだけ LLM を起動したかった。

**自作の理由**: 既存のものを使うより、自分で作った方が面白い。マネでもいい。Rust でバイブコーディングするのが好きなので、TypeScript 製の OpenClaw をそのまま使うより Rust で書き直す方が自然だった。社内チーム向けに閉じた用途なので、汎用性より業務フィット（Asana + Slack + GitHub）を優先できる。

## たまたまたどり着いた第3の設計

最初から「プログラムが主人、LLM は道具」という思想があったわけではない。OpenClaw をフォークするのではなく、一から全部自分で作ったら面白いし理解が深まるだろうと思って、一個ずつ積み上げていった。Symphony のポーリングを真似て、`claude -p` を subprocess で叩いて、Slack 連携を足して――そうやって作っていたら、**たまたまこうなった**。

振り返ると、世の中の AI エージェントは大きく2つの方向に進んでいる。OpenClaw は常時起動のセッションでエージェントが自律的に動く。[gstack](https://github.com/garrytan/gstack) は人間が `/plan-ceo-review` のようなコマンドで対話的に進める。

自分のは第3の設計になっていた。**プログラムが主人、LLM は道具**。

| | OpenClaw | gstack | ambient-task-agent |
|---|---|---|---|
| 主語 | エージェント | 人間 | **プログラム** |
| LLM の役割 | 自律判断 + 実行 | 対話的アシスタント | **呼ばれたら働く道具** |
| 待機中のコスト | セッション維持 | なし（手動起動） | **DB ポーリングだけ（LLM コスト 0）** |
| 制御方式 | LLM が次を判断 | 人間が判断 | **Rust が決定論的に判断** |

理由は3つ。

### 1. コスト予測可能性

LLM が「次に何をするか」を自律判断する設計では、コストが予測できない。ambient-task-agent は **タスクがあるときだけ `claude -p` を起動する**。待機中は純粋な DB クエリだけ。LLM 課金はタスク数に完全比例する。

### 2. 安全性

ステータス遷移を Rust のコードで制御している。「executing（実行中）のタスクは done / ci_pending / manual / conversing にしか遷移しない」がプログラムで保証される。LLM の気分で状態が変わることはない。

### 3. 暴走防止

`claude -p` は外部プロセスとして spawn される。10分タイムアウト + `kill_on_drop(true)` で、何があっても物理的に止まる。LLM が無限ループに入っても、Rust 側が殺す。

## 設計思想: Heartbeat + DB ポーリング — 不要なトークンを使わない

### よくある2つの方式の問題

**方式 A: `claude -p` を定期起動**

```bash
# 5分ごとに claude -p を起動してチェック
*/5 * * * * claude -p "Slackに新しい依頼があるか確認して、あれば処理して"
```

毎回 LLM を起動して「何もない」と判断するだけでトークンを消費する。1日中動かしたら、何もしていない時間にもサブスクのトークンが溶けていく。

**方式 B: tmux でセッション永続化**

```bash
# tmux 内で Claude Code を常駐させる
tmux new-session -d -s agent 'claude'
# → セッション内で /loop や対話的に待機させる
```

Claude Code のセッションを tmux で永続化して、常駐エージェントとして動かすアプローチ。セッションが生きている間はコンテキストが保持されるメリットがある。

ただし問題がある:
- **セッションが切れたら全ロスト** — tmux の crash やサーバー再起動で状態が消える
- **並列処理が難しい** — 1セッション = 1タスク。複数タスクを同時に処理できない
- **LLM が常時稼働** — 待機中もセッションが張られている
- **制御が曖昧** — 「このタスクは manual にして」のような状態遷移を外部から制御しにくい

### OpenClaw の Heartbeat をさらに押し進めた

[OpenClaw](https://github.com/openclaw/openclaw) は Heartbeat という仕組みを持っている。30分間隔で LLM がチェックリスト（`HEARTBEAT.md`）を読み込み、メール・カレンダー・通知などをまとめて1ターンで処理する。何もなければ `HEARTBEAT_OK` を返す。

賢い設計だが、**「何もない」と判断するのに毎回 LLM を呼んでいる**。

[Symphony](https://arxiv.org/abs/2506.01579)（Microsoft Research, 2025）はこれをさらに押し進めて、「プログラムが判断し、必要なときだけ LLM を呼ぶ」アーキテクチャを提案した。**「何かあるか？」の判断にLLM を使わない**。プログラムが DB をチェックして、処理対象があるときだけ LLM を起動する。

ambient-task-agent はこの Symphony のアプローチを真似た。Rust で **15秒間隔の DB ポーリング**を実装し、処理対象があるときだけ `claude -p` を起動する。「何かあるか？」は `SELECT` 文3本。LLM は一切関与しない。

```
┌──────────────────────────────────────────────────────────┐
│  Worker Heartbeat (15s)                                  │
│                                                          │
│  1. SELECT * FROM coding_tasks WHERE status IN (...)     │
│  2. SELECT * FROM ops_queue WHERE status = 'pending'     │
│  3. SELECT * FROM scheduled_jobs WHERE next_run_at < NOW │
│                                                          │
│  → 対象なし: sleep 15s（LLM コストゼロ）                    │
│  → 対象あり: classify → tokio::spawn で並列実行             │
└──────────────────────────────────────────────────────────┘
```

ポーリング自体は SQLite の `SELECT` 文3本。**LLM トークン消費ゼロ**。対象が見つかったときだけ `claude -p` を起動する。

### サブスク内で使い切る設計

Max プラン（月額固定）で運用している。つまり **LLM の呼び出し回数 = コスト** ではない。ただし、サブスクにもレートリミットがある。不要な呼び出しでレートリミットを消費したら、本当に必要なタスクが詰まる。

だから「LLM を呼ばない」ことが最大の最適化になる。

| | 定期 `claude -p` | tmux 常駐 | OpenClaw Heartbeat | **ambient-task-agent** |
|---|---|---|---|---|
| 待機中のトークン | 毎回消費 | セッション維持 | 毎回消費（1ターン） | **ゼロ** |
| 判断主体 | LLM | LLM | LLM（チェックリスト） | **プログラム**（`SELECT` 文） |
| チェック間隔 | cron 依存 | 任意 | 30分 | **15秒** |
| 並列実行 | 1プロセスずつ | 1セッション | キュー制御 | **tokio::spawn** |
| 状態永続性 | なし | tmux 依存 | DB | **SQLite** |
| 外部からの制御 | 困難 | 困難 | API 経由 | **DB 更新で即反映** |

### Rust が判断、LLM が作業

- **Rust 判定**: ステータス遷移、タイムアウト、優先度ソート、リトライ判定はすべてプログラムで決定
- **LLM 判定**: タスク分類（execute / converse）とタスク実行のみ LLM に委譲

LLM に判断させるのは「このタスクは明確か曖昧か」と「実際の作業」だけ。それ以外は全部プログラム。

## ステータスモデル: 決定論的な状態遷移

```
new → executing（明確）/ conversing（曖昧）
conversing → executing / manual / done / sleeping（5営業日タイムアウト）
manual → executing / done
executing → done / ci_pending / manual（stop） / conversing（ブロッカー検知）
```

これは Rust の `match` 文で制御されている。LLM の出力が何であっても、プログラムが許可した遷移しか起きない。

面白いのは **manual** ステータス。LLM が「これは自分では解決できない」と判断したとき、人間にバトンを渡す。人間が terminal で直接修正して「直した」と Slack に書くと、LLM が引き継ぐ。

## ops: 同僚の依頼をアンビエントに処理する

Slack チャンネルに投稿されたメッセージを自動で拾い、適切なスコープにルーティングして処理する。

```
同僚が Slack に書く
  ↓
LLM がメッセージ内容からスコープを分類
  ↓
該当スキルファイル（作業手順書）をシステムプロンプトに注入
  ↓
claude -p がスキルに従って作業実行
  ↓
結果を Slack スレッドに返信
```

**ルーティング**は2段階:
1. タスクキーで直接マッチ（DM 等チャンネルに紐づかないケース）
2. メッセージ内容で LLM が分類（コンテンツベースルーティング）

「この依頼はどのリポジトリの作業か？」を LLM に判定させて、該当するスキルファイルを読み込んで実行する。スコープ外のメッセージは「対応不要」として弾く。

### Inception モード: 曖昧な依頼を2ターンで要件定義

「〇〇作って」のような曖昧な依頼には、2ターンの要件定義フローがある。

```
ターン1: LLM が依頼意図を分析して質問を生成 → Slack に投稿
ターン2: ユーザーが回答 → LLM が要件を整理してタスク分解
  → 承認ボタン付きで Slack に投稿
  → ユーザーが「承認」→ 自動実行
```

## スケジューラ: PM スキル

タスク自動化だけでなく、PM 的な仕事も自動化している。

| ジョブ | 内容 |
|--------|------|
| 朝会ブリーフィング | 当日タスク一覧 + Google Calendar の空き時間にタスクを自動配置（タイムボクシング） |
| 夕会サマリー | 進捗まとめ |
| 停滞チェック | 長時間動いていないタスクを検知して通知 |
| 自己改善 | 分類精度を分析して改善タスクを提案 |

朝会では Google Calendar と連携して、空き時間にタスクを自動配置する。カレンダーにタスクブロックが登録されるので、1日の計画が自動で立つ。

## このハーネスの確信があったから RAG を捨てられた

前回の記事で書いたナレッジ Bot の話に戻る。

`claude -p` に `--allowedTools "Read,Grep,Glob"` を渡して、ナレッジディレクトリの中を自由に検索させる。RAG パイプライン（Embedding、ベクトル DB、リランキング）は一切不要。

この判断ができたのは、ambient-task-agent で毎日 `claude -p` を使い倒していたからだ。

- `claude -p` は指示通りにファイルを検索・読み込み・統合して回答を作れる
- `--max-turns` で暴走を防止できる
- `--append-system-prompt` で役割と制約を定義できる
- `kill_on_drop` でタイムアウト時にプロセスを確実に殺せる

**ハーネスが信頼できる**と知っていたから、RAG パイプラインを丸ごと捨てる判断ができた。知らなかったら Dify を使い続けていたと思う。

## claude -p のハーネスは進化し続けている

`claude -p` は毎週のようにアップデートされている。最近追加された機能だけでも:

- `--fallback-model`: Opus が過負荷のときに自動で Sonnet にフォールバック
- `--json-schema`: 構造化出力（ルーティングの分類に使っている）
- `--max-budget-usd`: API 利用時の予算上限
- `--resume`: セッション継続（ナレッジ Bot の更問いに使っている）

ハーネスが進化するたびに、自分のコードを変えなくても機能が増える。**乗っているだけで恩恵を受ける**。これが自前で LLM ループを実装するのとの決定的な違い。

## トレードオフ: 正直に書く

### stop でプロセスが即停止しない

`claude -p` は外部プロセスなので、「stop」コマンドを受けても実行中のプロセスは走り切る。DB のステータスを `manual` に変えて結果を破棄するが、トークンは消費される。本質的な解決には `claude -p` 側にキャンセル API が必要。

### コンテンツベースルーティングの誤分類

LLM によるメッセージ分類は完璧ではない。スコープ外のメッセージが誤って特定のスキルにルーティングされることがある。スキルファイルに「Gotchas（よくある失敗）」セクションを追加して精度を上げている。

### コードが見せられない

社内チーム向けに作ったツールなので、コードは非公開。設計思想とアーキテクチャを共有することで、同じアプローチを取りたい人の参考になれば幸い。

## 結論: ハーネスに乗れ

自律エージェントが注目を集めている。LLM が自分で考え、自分で判断し、自分で動く。それは確かにかっこいい。

でも、チーム業務の自動化には別のアプローチが合っていた。

- **コストを予測したい** → タスク数に比例する設計
- **暴走を止めたい** → プログラムが制御する設計
- **チームの依頼を捌きたい** → アンビエントに動く設計

LLM は道具。プログラムが主人。ハーネスが優秀なら、道具は十分に働く。

前回の記事で「エージェンティックサーチを信じろ」と書いた。今回はこう書く。

**ハーネスに乗れ。** `claude -p` の進化に乗るだけで、自律エージェントを作らなくても十分に戦える。
