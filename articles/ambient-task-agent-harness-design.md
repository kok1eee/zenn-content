## はじめに

> 業務自動化Pythonエンジニア → Rust でバイブコーディングに目覚めた人。

自分で RAG チャットボットをいくつか作ると「なんかイマイチだな」と思う瞬間がある。チャンク分割のせいで文脈が切れる。ベクトル検索で関係ないドキュメントが上位に来る。LLM 自体は賢いのに、自分が作ったパイプラインが足を引っ張っている。

一方で Claude Code を毎日使っていると、こいつは Grep と Read でファイルを直接読んで、複数ファイルを統合して回答を作れる。RAG パイプラインって要る？

https://zenn.dev/tazawa_masayoshi/articles/knowledge-chatbot-agentic-search

前回の記事ではその話を書いた。でもナレッジ Bot は副産物だった。本当のきっかけはもっと手前にある。

自分たちの日常を見ていて気づいた。同僚から「これやって」と Slack が来たら、自分は Claude Code を開いて、依頼内容をそのまま貼り付けて、結果を Slack に返している。**自分を挟む必要がないのでは？** ……と最初は思った。実際にはそんなに単純ではなかった（後述する）。

だったら Slack のメッセージを直接 `claude -p` に渡して、結果を Slack に返せばいい。自分が寝ていても動く仕組みにすればいい。`claude -p` を使えばサブスクの範囲で動くし、ツール制限やサンドボックスを自分で作らなくて済む。

実際に今やっていることは、Slack 経由でコーディングしてもらう感覚に近い。同僚から依頼が来たら、設定ファイルを書き換えて push したり、定型作業を一時対応してもらったり。PR を出すような大きな仕事もあるけど、日常的には「ちょっとこれやっておいて」を捌いてくれるやつ。

そうやって作り始めたのが ambient-task-agent。作っていく中で、設計の関心事が**プロンプト → コンテキスト → ハーネス → アンビエント**の4つに分類できることに気づいた。順番に登ったというより、同時並行で必要に迫られて積み上げていった。

:::message
社内チーム向けに作ったツールのため、コードは非公開。実装は Rust + SQLite + Slack Socket Mode だが、この記事ではコードではなく設計思想とアーキテクチャの話をする。「`claude -p` を使って何かを自動化したい」人向け。言語やスタックは問わない。
:::

## LLM を使ったシステム設計の進化

LLM を使ったシステムの設計は、この数年で階層が増えてきた。

### プロンプトエンジニアリング — LLM に何を伝えるか

一番最初のフェーズ。指示の書き方を工夫する。ロールを設定する。few-shot で例を見せる。ChatGPT が出てきた頃、みんなここに集中していた。

### コンテキストエンジニアリング — LLM にどんな文脈を渡すか

プロンプトだけでは足りないと気づく。RAG で関連ドキュメントを検索して渡す。メモリで過去の会話を保持する。「何を伝えるか」から「どんな文脈の中で考えさせるか」にシフトした。

### ハーネスエンジニアリング — LLM をどう制御するか

LLM にツールを持たせてコードを書かせるようになると、制御が必要になった。ツール制限、状態管理、暴走防止、多層検証。Claude Code はまさにこのハーネスを提供するプロダクト。`--allowedTools` でツールを制限し、`--max-turns` で暴走を防ぎ、サンドボックスで安全に実行する。

https://qiita.com/miruky/items/155f3b5a0dcde72fcd10

https://note.com/tasty_dunlin998/n/na32378cf15c7

最近「ハーネスエンジニアリング」という言葉を見かけるようになった。後者の記事では「交代可能な実行器を、止められ、追え、つなぎ直せる形で、現実の仕事に埋め込むこと」と表現されている。

ちなみに自分は Claude Code のハーネスの上にさらに制御構造を積む [o-m-cc](https://github.com/kok1eee/o-m-cc) というプラグインも作っている。並列レビュー Council、品質ゲート、セッション引き継ぎなど、ハーネスエンジニアリングの実践をプラグインとして配布しているもの。

### アンビエントエンジニアリング — LLM をどう環境に溶け込ませるか

ハーネスが揃っても「人間が起動する」なら普通のツール。次の段階は、LLM を環境に溶け込ませること。常駐し、イベントを拾い、チームの仕事に勝手に組み込まれる。人間が意識しなくても動いている。

[OpenClaw](https://github.com/openclaw/openclaw) はこの段階にいる。Heartbeat で常駐し、マルチチャンネルでイベント駆動で動く。

| 段階 | 問い | 代表例 |
|------|------|--------|
| **プロンプト** | 何を伝えるか | CLAUDE.md、プロンプトテンプレート |
| **コンテキスト** | どんな文脈を渡すか | RAG、メモリ、few-shot |
| **ハーネス** | どう制御するか | Claude Code、gstack |
| **アンビエント** | どう環境に溶け込ませるか | OpenClaw、ambient-task-agent |

自分のシステムは一個ずつ積み上げていたら、たまたまこの4段階を全部踏んでいた。ここからは、各段階で自分が何をやったかを書く。

## 作ったもの: ambient-task-agent

先に全体像を見せる。

```
入口が違うだけで、中の処理は同じ

  Slack メッセージ ─┐
                    ├→ classify → 明確 → そのまま実行 → PR
  Asana タスク    ─┘            → 曖昧 → Slack ラリーで要件確定 → 実行
                                → 詰まった → manual（人間が terminal で対応）
```

| 入口 | 処理 |
|------|------|
| Slack ops（質問・依頼） | 自動回答 or 要件ヒアリング → タスク昇格 |
| Slack ops（定型作業） | スキル実行 → Slack 返信で完了 |
| Asana タスク | 分類 → 実行 → PR → CI |

コーディングだけでなく、PM スキル（朝会ブリーフィング、タイムボクシング、停滞検知、Google Calendar 連携）も備えている。

## なぜ OpenClaw を使わなかったか

最初に検討したのは [OpenClaw](https://github.com/openclaw/openclaw)。マルチチャンネル対応の汎用パーソナルアシスタントとして完成度が高い。でも、チーム業務自動化という用途では合わなかった。

まずセキュリティ。OpenClaw はデフォルトでエージェントがホスト上でフルアクセスで動作する。チーム Slack からの任意メッセージがコード実行に繋がる経路を持つ。自分たちの用途では、入力を Asana タスク（承認済み）と ops 管理者に限定したかった。攻撃面が広すぎる。

次にコスト。OpenClaw は Gateway が常時起動し、セッションが持続する。Heartbeat も30分ごとに LLM を呼ぶ。タスクがあるときだけ LLM を起動したかった。

あと、単純に自分で作った方が面白い。マネでもいい。Rust でバイブコーディングするのが好きなので、TypeScript 製の OpenClaw をそのまま使うより Rust で書き直す方が自然だった。社内チーム向けに閉じた用途なので、汎用性より業務フィット（Asana + Slack + GitHub）を優先できる。

OpenClaw をフォークするのではなく、一から全部自分で作ったら理解が深まるだろうと思って、一個ずつ積み上げていった。

でもチームの Slack に繋いだ瞬間、ガードレールが必要になった。誰でもメッセージを送れるからセキュリティが要る。24時間動かすからコスト制御が要る。止められないと困るから状態管理が要る。一個ずつ足していたら、気づいたらプログラムがほとんど全部を制御していた。LLM がやるのは「どのツールを使って作業するか」だけ。

振り返ると、4つのエンジニアリングの階段を自然に登っていた。

## 第1段階: プロンプトエンジニアリング — LLM に何を伝えるか

最初に取り組んだのは、`claude -p` に渡す指示の最適化。

### 設定ファイル + 作業手順書で「何をさせるか」を定義する

`claude -p` は Claude Code のネイティブ Skills を読み込まない。だから自前で仕組みを作った。

まず toml の設定ファイルで、各リポジトリの作業スコープを定義する。

```toml
[[repo]]
key = "hikken_schedule"
ops_description = "必見パチ/スロの「▼サブカテ追加」「▼クローズ対応」指示のみ"
ops_skills = [".claude/skills/ops/SKILL.md"]
```

`ops_skills` で指定したマークダウンファイルが作業手順書。Rust がこれを読み込んで、同フォルダの `gotchas.md` と `references/` も自動結合し、`claude -p` の `--append-system-prompt` に注入する。

```
.claude/skills/ops/
├── SKILL.md          # 作業手順
├── gotchas.md        # よくある失敗（自動読み込み）
└── references/       # 詳細リファレンス（自動読み込み）
```

これで2つに分かれる:

- **手順書がある** → ノンストップで自動処理。設定変更、ファイル更新、push まで一気通貫
- **手順書がない** → 分析・回答だけして、人間に判断を返す

**何を自動化するかは、手順書を書くかどうかで決まる**。

### Gotchas セクションが一番効く

スキルファイルで一番効果があるのは Gotchas（よくある失敗）セクション。

```markdown
## Gotchas
- チャンネル内の全メッセージが作業対象ではない。「▼サブカテ追加」の指示マーカーがあるものだけ
- 障害報告・停止報告・確認依頼は ops スコープ外。「対応不要」として返すこと
- image_mappings.yaml の編集時は既存エントリの順序を崩さない
```

LLM は賢いけど、ドメイン固有の「やってはいけないこと」は教えないとわからない。運用しながら失敗パターンを蓄積していく。

## 第2段階: コンテキストエンジニアリング — LLM にどんな文脈を渡すか

プロンプトの次は「何を一緒に渡すか」。

### 会話履歴の管理

ops のスレッド返信は会話として蓄積される。同じスレッドで追加指示が来たら、過去の会話を含めて `claude -p` に渡す。セッションをまたぐ会話が可能。

### 完了記録と学習蓄積

タスクが完了するたびに `context.md`（完了記録）と `memory.md`（学習メモ）に蓄積する。次のタスク実行時に参照できる。

### few-shot 分類学習

タスクが「明確（即実行）」か「曖昧（要確認）」かの分類に、過去の分類履歴を few-shot で渡している。正解/不正解の記録が蓄積されるほど、分類精度が上がる。

## 第3段階: ハーネスエンジニアリング — LLM をどう制御するか

ここが一番重要で、一番時間をかけた。

### claude -p のハーネスは強い

`claude -p` はただのヘッドレスモード CLI ではない。

- `--append-system-prompt`: 役割と制約を定義
- `--allowedTools`: 使えるツールを制限
- `--max-turns`: ターン数制限で暴走防止

そして Rust 側から `kill_on_drop(true)` で10分タイムアウト。何があっても物理的に止まる。ツール制限・暴走防止・タイムアウトを自分で実装しなくていい。`claude -p` が全部やってくれる。

### Heartbeat + DB ポーリング

LLM を常駐エージェントとして使う一般的な方法は「tmux でセッション永続化」か「cron で `claude -p` を定期起動」。でもどちらも待機中にトークンを消費する。

[OpenClaw](https://github.com/openclaw/openclaw) は Heartbeat という仕組みを持っている。30分間隔で LLM がチェックリストを読んで処理する。賢い設計だが、**「何もない」と判断するのに毎回 LLM を呼んでいる**。

ambient-task-agent は DB ポーリングで必要なときだけ LLM を起動する設計にした。後から [Symphony](https://arxiv.org/abs/2506.01579)（Microsoft Research, 2025）を見つけて「同じことを言っている」と気づいた。論文を先に読んで設計したわけではない。Rust で **15秒間隔の DB ポーリング**を実装し、処理対象があるときだけ `claude -p` を起動する。

```
┌──────────────────────────────────────────────────────────┐
│  Worker Heartbeat (15s)                                  │
│                                                          │
│  1. SELECT * FROM coding_tasks WHERE status IN (...)     │
│  2. SELECT * FROM ops_queue WHERE status = 'pending'     │
│  3. SELECT * FROM scheduled_jobs WHERE next_run_at < NOW │
│                                                          │
│  → 対象なし: sleep 15s（LLM コストゼロ）                    │
│  → 対象あり: classify → tokio::spawn で並列実行             │
└──────────────────────────────────────────────────────────┘
```

「何かあるか？」は `SELECT` 文3本。LLM は一切関与しない。

| | 定期 `claude -p` | tmux 常駐 | OpenClaw Heartbeat | **ambient-task-agent** |
|---|---|---|---|---|
| 待機中のトークン | 毎回消費 | セッション維持 | 毎回消費（1ターン） | **ゼロ** |
| 判断主体 | LLM | LLM | LLM（チェックリスト） | **プログラム**（`SELECT` 文） |
| チェック間隔 | cron 依存 | 任意 | 30分 | **15秒** |
| 並列実行 | 1プロセスずつ | 1セッション | キュー制御 | **tokio::spawn** |
| 状態永続性 | なし | tmux 依存 | DB | **SQLite** |
| 外部からの制御 | 困難 | 困難 | API 経由 | **DB 更新で即反映** |

`claude -p` は Claude Code のサブスク（Max プラン）の範囲で使える。API 課金ではないので、タスクが増えても月額は変わらない。ただしサブスクにもレートリミットがあるので、不要な呼び出しでレートリミットを消費したら本当に必要なタスクが詰まる。だから「LLM を呼ばない」ことが最大の最適化になる。

### 決定論的な状態遷移

```
new → executing（明確）/ conversing（曖昧）
conversing → executing / manual / done / sleeping（5営業日タイムアウト）
manual → executing / done
executing → done / ci_pending / manual（stop） / conversing（ブロッカー検知）
```

Rust の `match` 文で制御。LLM の出力が何であっても、プログラムが許可した遷移しか起きない。

面白いのは **manual** ステータス。LLM が「自分では解決できない」と判断したとき、人間にバトンを渡す。人間が terminal で直接修正して「直した」と Slack に書くと、LLM が引き継ぐ。

### 多層フィードバック

`claude -p` の出力は多層で検証される。

1. **clippy + cargo test** — コンパイル時チェック
2. **git-ratchet** — テスト数が減ったり warnings が増えたら PR 拒否
3. **CI 自動リトライ** — CI が落ちたら原因を分析して自動修正
4. **self_improvement** — 週次で分類精度・エラーパターンを分析して改善提案

## 第4段階: アンビエントエンジニアリング — 環境に溶け込ませる

プロンプト・コンテキスト・ハーネスが揃っても、「人間が起動する」なら普通のツールでしかない。

**アンビエント**とは、誰かが Slack に書いたら、自分が何もしなくても勝手に動くこと。

```
同僚「この設定追加しておいて」
  ↓
（自分は何もしていない）
  ↓
Bot「完了しました。設定ファイルを更新して反映済みです」
```

本人は別の仕事をしている。会議中かもしれない。寝ているかもしれない。それでも依頼は処理される。

OpenClaw や gstack は「自分が Claude に指示して、自分のタスクを進める」ツール。ambient-task-agent は違う。**チームの依頼を勝手にさばくシステム**。主語が自分ではなく同僚。だからアンビエント。

### ops: Slack からのルーティング

```
同僚が Slack に書く
  ↓
LLM がメッセージ内容からスコープを分類
  ↓
該当スキルファイルをシステムプロンプトに注入
  ↓
claude -p がスキルに従って作業実行
  ↓
結果を Slack スレッドに返信
```

### Inception モード: 曖昧な依頼を2ターンで要件定義

「〇〇作って」のような曖昧な依頼には、2ターンの要件定義フローがある。

```
ターン1: LLM が依頼意図を分析して質問を生成 → Slack に投稿
ターン2: ユーザーが回答 → LLM が要件を整理してタスク分解
  → 承認ボタン付きで Slack に投稿
  → ユーザーが「承認」→ 自動実行
```

### PM スキル

タスク自動化だけでなく、PM 的な仕事もアンビエントに。

| ジョブ | 内容 |
|--------|------|
| 朝会ブリーフィング | 当日タスク一覧 + Google Calendar の空き時間にタスクを自動配置 |
| 夕会サマリー | 進捗まとめ |
| 停滞チェック | 長時間動いていないタスクを検知して通知 |
| 自己改善 | 分類精度を分析して改善タスクを提案 |

## このハーネスの確信があったから RAG を捨てられた

前回の記事で書いたナレッジ Bot の話に戻る。

`claude -p` に `--allowedTools "Read,Grep,Glob"` を渡して、ナレッジディレクトリの中を自由に検索させる。RAG パイプライン（Embedding、ベクトル DB、リランキング）は一切不要。

この判断ができたのは、ambient-task-agent で毎日 `claude -p` を使い倒していたからだ。

- `claude -p` は指示通りにファイルを検索・読み込み・統合して回答を作れる
- `--max-turns` で暴走を防止できる
- `--append-system-prompt` で役割と制約を定義できる
- `kill_on_drop` でタイムアウト時にプロセスを確実に殺せる

**ハーネスが信頼できる**と知っていたから、RAG パイプラインを丸ごと捨てる判断ができた。知らなかったら Dify を使い続けていたと思う。

## `claude -p` は進化し続けている

`claude -p` は毎週のようにアップデートされている。最近追加された機能だけでも:

- `--fallback-model`: Opus が過負荷のときに自動で Sonnet にフォールバック
- `--json-schema`: 構造化出力
- `--max-budget-usd`: API 利用時の予算上限
- `--resume`: セッション継続
- `--bare`: hooks・プラグイン・スキル読み込みを全スキップ。スクリプトから叩くときの起動が軽くなる

Claude Code が進化するたびに、自分のコードを変えなくても機能が増える。**乗っているだけで恩恵を受ける**。これが自前で LLM ループを実装するのとの決定的な違い。

裏を返せば、Anthropic が breaking change を入れたら自分の制御構造が壊れるリスクがある。ハーネスに乗るとはハーネスに依存すること。そのリスクは受け入れている。もし `claude -p` が使えなくなったら [Strands Agents](https://github.com/strands-agents/sdk-python) のような別のハーネスに乗り換える。主導権をプログラムに持たせている設計なら、ハーネスの差し替えは難しくない。

## トレードオフ: 正直に書く

### stop でプロセスが即停止しない

`claude -p` は外部プロセスなので、「stop」コマンドを受けても実行中のプロセスは走り切る。DB のステータスを `manual` に変えて結果を破棄するが、トークンは消費される。

### コンテンツベースルーティングの誤分類

LLM によるメッセージ分類は完璧ではない。スコープ外のメッセージが誤って特定のスキルにルーティングされることがある。スキルファイルに Gotchas セクションを追加して精度を上げているが、症状への対処であって根本解ではない。

### 「自分を挟む必要がない」は嘘だった

冒頭で「自分を挟む必要がないのでは？」と書いた。正直に言うと、半分嘘。

自分が中継していたときは、依頼の意図を暗黙に検証していた。「この依頼はこういう意味だな」「この作業はこのリポジトリだな」を人間が判断していた。エージェントに任せると、その暗黙の検証がなくなる。

だから `ops_admin` 制限がある。誰でも依頼できるわけではなく、管理者からのメッセージだけ処理する。定型作業（スキルファイルに手順が書かれたもの）は安心して任せられるが、曖昧な依頼は Inception モードで要件確認を挟む。完全な無人化ではなく、**任せられるところだけ任せる**。

実際、誤分類でスコープ外のメッセージを処理しかけたことはある。任せて全部 OK ではない。Gotchas を育てて、誤分類のパターンを self_improvement で分析して、アウトプットの精度を上げ続けている。その上で、一時対応や簡単な定型作業はノンストップでやってもらう。チューニングしながら委任範囲を広げていく、という運用。

### コストは「LLM 無料」だけじゃない

`claude -p` のサブスクで LLM コストはゼロと書いたが、システム全体の維持コストはゼロではない。

- レートリミット枯渇で本来のタスクが詰まる可能性
- Gotchas を人間が蓄積・メンテナンスするコスト
- 誤実行が起きたときのリカバリーコスト

「LLM コストゼロ」はポーリング部分の話であって、運用全体のコストとは別の話。

## 何をハーネスに任せ、何を自分で設計したか

振り返ると、判断の連続だった。

**ハーネス（Claude Code）に任せたもの:**
- ツール制限（`--allowedTools`）
- サンドボックス実行
- セッション管理（`--resume`）
- 暴走防止（`--max-turns`）

**自分で設計したもの:**
- いつ LLM を起動するか（DB ポーリング、Symphony のアプローチ）
- 何を処理するか（コンテンツベースルーティング）
- いつ止めるか（ステータスモデル、`kill_on_drop`）
- 誰に任せるか（ops_admin 制限）
- 失敗からどう学ぶか（self_improvement、Gotchas 蓄積）

この境界線が一番大事だった。ハーネスが得意なことはハーネスに任せる。でも「いつ・何を・いつまで」の判断はプログラムでやる。ここを LLM に任せると、コストも安全性も予測できなくなる。

## 次の課題

4段階を登ってアンビエントまで来た。まだ育てている途中で、見えている課題がある。

- **複数エージェントの協調** — 今は1タスク = 1 `claude -p`。タスクが複雑化したら分割・並列実行が必要になる
- **委任スコープの明示** — 「どこまでエージェントに任せるか」を組織レベルで設計する。技術の問題ではなくガバナンスの問題
- **信頼の設計** — エージェントがやった判断を、どうやって事後検証するか。ログと監査の仕組み

プロンプトエンジニアリングで指示を書く。コンテキストエンジニアリングで文脈を渡す。ハーネスエンジニアリングで制御する。アンビエントエンジニアリングで環境に溶け込ませる。一個ずつ積み上げた結果、ここまで来た。

前回の記事で「エージェンティックサーチを信じろ」と書いた。今回はこう書く。

**ハーネスに乗れ。その上に制御構造を自分で積め。そして、LLM に主導権は渡すな。**

Claude Code というハーネスは Anthropic の天才たちが毎週進化させている。`claude -p` でそれをコードから呼べる。自分でエージェントフレームワークを作る必要はない。ツール制限・サンドボックス・セッション管理はハーネスに任せればいい。

でも「いつ起動するか」「何を処理するか」「いつ止めるか」「誰に任せるか」——これはプログラムの仕事だ。ここを LLM に委ねた瞬間に、コストも安全性も予測できなくなる。

ハーネスに乗って、環境を自分で設計して、主導権はプログラムに持たせる。それが今のところの答え。
